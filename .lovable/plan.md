

## הבעיה

הנכס `מרסל ינקו 10` ב-yad2 שמור עם `features: {parking: true}` בלבד וסטטוס `backfill_status: not_needed` — **למרות שבדף יד2 המקורי יש המון פיצ'רים** (מיזוג, מעלית, סורגים, משופץ, מרפסת, ממ"ד, מרוהט, מחסן וכו').

## הסיבה השורשית

ב-`supabase/functions/_shared/property-helpers.ts` (שורות 532-540), הקוד שמחליט אם נכס צריך השלמת נתונים בודק:

```ts
property.features && Object.keys(property.features).length > 0
```

זאת אומרת: **אם הסקראנר הראשוני חילץ ולו פיצ'ר אחד** (אפילו `false`, אפילו רק `parking`) — הנכס מסומן כ-`not_needed` והבאקפיל לעולם לא ירוץ עליו כדי להשלים את השאר.

הסקראנר של יד2 (Jina/list view) רואה רק parking בכרטיסי הסיכום — כל שאר הפיצ'רים נמצאים רק בעמוד הפנימי של ההודעה. לכן כל נכסי יד2 שיש להם רק `{parking: ...}` תקועים בלי השלמה.

## הפתרון המוצע

### 1. החמרת תנאי "יש מספיק features" — דרישת מינימום סביר
בקובץ `_shared/property-helpers.ts`, להחליף את הבדיקה הנאיבית בלוגיקה שדורשת:
- מינימום **3 מפתחות פיצ'רים** (לא רק אחד), **או**
- שלפחות אחד מהפיצ'רים המשמעותיים נוכח (`mamad`, `elevator`, `aircon`, `balcony`, `furnished`, `renovated`)

נכס שיש לו רק `{parking}` → ייחשב חסר → ימוקם בתור ל-backfill.

### 2. אפס את הנכסים ה"תקועים" שכבר במצב הזה
שאילתה חד-פעמית ב-Edge Function שמרצים אותה פעם אחת:
- כל הנכסים מ-yad2 שיש להם `backfill_status='not_needed'` ו-features עם פחות מ-3 מפתחות → reset ל-`pending`.
- זה יחזיר כ-מאות-אלפי נכסים יד2 לתור הבאקפיל וההשלמה הבאה תמלא להם את שאר הפיצ'רים מהעמוד הפנימי.

### 3. חשיפת כפתור "אפס נכסי features חלשים" ב-UI (אופציונלי)
בכרטיסיית "השלמת נתונים" אפשר להוסיף כפתור משני נוסף: "אפס נכסים עם פיצ'רים דלים" — בדומה לכפתור "נסה שוב נכשלים" שכבר נבנה. ככה אפשר להריץ את האיפוס שוב בעתיד אם הסף משתנה.
