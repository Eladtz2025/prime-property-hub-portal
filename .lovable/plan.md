
לא, לא בדיוק SQL ישיר. אני מסביר מה בדיוק יקרה:

## מה אני מריץ — בפועל

**קריאה ל-Edge Function** ששמה `backfill-property-data-jina` עם פרמטר `action: 'reset_weak_features'`.

ה-Edge Function הזה (קוד שכבר קיים מהפעם הקודמת) עושה בתוכו פעולת UPDATE על הדאטאבייס — שזה כן SQL, אבל לא SQL שאני כותב ישירות ומריץ. זה קוד שכבר קיים, מוגדר, ובטוח.

## למה לא SQL ישיר?

1. **ביטחון** — ה-Edge Function עושה את הסינון המדויק (yad2 בלבד, פעיל, not_needed, פחות מ-3 פיצ'רים, ללא פיצ'ר משמעותי) בקוד שאפשר לבדוק ולתחזק. SQL ישיר יהיה שורה ארוכה ומסוכנת יותר.
2. **חזרתיות** — אם בעתיד תרצה לאפס שוב (למשל אחרי שהסף ישתנה), ה-Edge Function כבר מוכנה.
3. **לוגים** — הקריאה תופיע בלוגים של ה-Edge Function כך שנוכל לראות בדיוק כמה נכסים אופסו.

## מה ההבדל בפועל?

| | SQL ישיר | קריאה ל-Edge Function |
|---|---|---|
| מה רץ | `UPDATE scouted_properties SET backfill_status='pending' WHERE...` | אותו UPDATE, אבל בתוך פונקציה |
| מי כותב את זה | אני, עכשיו | קוד שכבר קיים ונבדק |
| לוגים | אין | יש |
| חזרתיות | חד-פעמי | אפשר להריץ שוב |

בשני המקרים: ~237 נכסי yad2 יעברו מ-`not_needed` ל-`pending`, והריצה הלילית הבאה תמלא להם פיצ'רים.
